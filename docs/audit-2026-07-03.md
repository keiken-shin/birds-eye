## P0 — Security

### 1. No Content-Security-Policy

**What:** `src-tauri/tauri.conf.json` sets `"csp": null`.

**Why it matters:** the webview renders **untrusted input everywhere** — file and folder names from the scanned disk. Today React escapes them, but a single future `dangerouslySetInnerHTML`, a markdown renderer, or a third-party component turns a malicious filename (`<img src=x onerror=...>.mp4` is a legal NTFS name) into script execution *inside a window that can invoke every Tauri command* — including cleanup-plan execution. CSP is the second wall when output-escaping fails; production apps need both.

**Impact:** XSS → full command surface → silent deletion of user files (recycle bin softens it, but a 30-day-quiet attacker still wins).

**Fix:**

```jsonc
// src-tauri/tauri.conf.json → app.security
"csp": "default-src 'self'; img-src 'self' asset: data:; style-src 'self' 'unsafe-inline'; font-src 'self' https://fonts.gstatic.com; connect-src 'self' ipc: http://ipc.localhost"
```

(`style-src 'unsafe-inline'` is needed for the inline style props; tighten later by moving to classes. Verify the exact `connect-src` Tauri v2 needs on Windows — `http://ipc.localhost` — in dev tools.)

### 2. Asset protocol scoped to the entire filesystem

**What:** `"assetProtocol": { "enable": true, "scope": ["**"] }` — and the current frontend never uses `convertFileSrc` at all.

**Why:** any code running in the webview can read *any file the user can read* via `asset://` URLs — SSH keys, browser profiles, tax documents. It exists to serve the old frontend's media preview, which was deleted.

**Impact:** turns any XSS (see #1) from "app-scoped" into "whole-disk read".

**Fix now:** disable it. **Fix when file preview returns (M10):** re-enable scoped to scanned roots only:

```jsonc
"assetProtocol": { "enable": false, "scope": [] }
// M10, per-root: register scopes at runtime with the fs-scope API instead of "**"
```

## P1 — Integrity & reliability

### 3. `trash_files` takes arbitrary paths — and nothing calls it

**What:** `api.rs::trash_files` recycles any list of paths sent from the webview, no containment check. Compare `delete_index`, which correctly canonicalizes and enforces the app-dir prefix. The workspace UI never invokes it (the cleanup path goes through plans, which re-verify against the ontology predicate).

**Impact:** free primitive for webview-side code to disappear arbitrary user files.

**Fix:** remove `trash_files` (and `scan_to_index`, also unused) from `generate_handler!` and delete the bridge exports. Attack surface you don't register can't be abused. If a raw-trash path is ever needed again, validate each path is a live `files.path` row in a known index first.

### 4. Executor can trash a file and lose the restore record

**What:** in `executor.rs::execute_plan_with`, the sequence per candidate is: `send_to_trash()` → `INSERT cleanup_log ?` → `UPDATE files ?`. The two DB writes propagate errors with `?`. If the log insert fails *after* a successful trash (realistic: the executor opens a **raw** connection with no busy-timeout — see #5 — so a concurrent enrichment write can produce `database is locked`), the whole function aborts: the file is in the recycle bin, but the Library has no entry, the Undo toast never lists it, and the remaining candidates are silently skipped while the plan stays `draft`.

**Impact:** the app's central promise — "everything cleaned is one click from restored" — silently broken exactly under the concurrency the app itself creates.

**Fix:** write the log row *before* trashing with status `pending`, promote it after; treat DB errors with the same per-file isolation as trash errors:

```rust
for cand in &candidates {
    let log_id = insert_log(conn, cand, "pending", expires_at)?;      // before any side effect
    match trasher.send_to_trash(Path::new(&cand.path)) {
        Ok(()) => {
            if let Err(e) = promote_log(conn, log_id, "in_recycle_bin")
                .and_then(|_| mark_deleted(conn, cand.file_id, now))
            {
                failed.push(CleanupFailure { file_id: cand.file_id, path: cand.path.clone(),
                    reason: format!("trashed but bookkeeping failed: {e}") });
                continue; // file is still recoverable via the pending log row
            }
            cleaned += 1; bytes_cleaned += cand.size.max(0) as u64;
        }
        Err(reason) => { let _ = delete_log(conn, log_id); failed.push(/* … */); }
    }
}
```

### 5. Connection acquisition is inconsistent (23+ raw opens)

**What:** the busy-timeout + WAL fix lives only in `IndexWriter::open`. `api.rs` (23 sites), `executor.rs`, `restore.rs`, `orchestrator.rs`, and `jobs.rs` call `Connection::open` directly — no busy timeout, so any lock contention during a scan/enrichment write returns `database is locked` immediately on those paths. This is also a DRY violation: connection policy is a decision, and it's currently made in one place and ignored in twenty-four.

**Fix:** one acquisition function, used everywhere:

```rust
// src/index/mod.rs
pub fn open_index_connection(path: &Path) -> rusqlite::Result<Connection> {
    let conn = Connection::open(path)?;
    conn.busy_timeout(Duration::from_secs(5))?;
    let _: String = conn.query_row("PRAGMA journal_mode=WAL", [], |r| r.get(0))?;
    Ok(conn)
}
```

Then mechanical replacement of every `Connection::open(` outside tests. (`IndexWriter::open` calls it too.)

### 6. Unbounded job-event buffers, jobs never evicted

**What:** `ScanJobManager.jobs: HashMap<u64, JobState>` grows monotonically; each `JobState.events: Vec<JobEventDto>` grows with every progress tick and log line for the life of the process. A 734k-file scan emits tens of thousands of events, each carrying heap strings.

**Impact:** long sessions with several scans leak tens-to-hundreds of MB; the `job_events_since` clone (`[offset..].to_vec()`) amplifies it.

**Fix:** cap per-job history and evict terminal jobs:

```rust
const MAX_EVENTS_PER_JOB: usize = 1_000;   // UI shows last 200 lines anyway
if job.events.len() >= MAX_EVENTS_PER_JOB { job.events.remove(0); } // or VecDeque
job.events.push(event.clone());
// on terminal status: retain only the last 3 finished jobs in the map
```

The frontend already tolerates missing early events (it backfills from offset 0 but only displays the tail).

### 7. ✅ Retention expiry was never invoked

**What:** `expire_old_entries` existed with tests but had no production caller, so "recoverable for 30 days" was enforced in neither direction — entries stayed `in_recycle_bin` forever and restore attempts after the OS bin was emptied produced a confusing per-item error.

**Fixed:** lazy expiry on every Library read (`recently_cleaned_log` now calls it). Cheap, no scheduler needed.

## P2

### 8. Explorer argument built by string interpolation

**What:** `command.raw_arg(format!("/select,\"{explorer_path}\""))`. A path containing `"` (legal in some contexts, and attacker-influenced via file names) breaks the quoting and injects additional arguments into `explorer.exe`'s command line.

**Impact:** low (explorer args, not a shell), but it's an injection primitive in a code path that exists purely for convenience.

**Fix:** reject paths containing `"` before formatting, or better, use the documented no-quotes form: `command.raw_arg(format!("/select,{explorer_path}"))` after verifying it handles spaces (it does — explorer splits on the comma, not spaces), or shell out via `SHOpenFolderAndSelectItems` through the `windows` crate for a string-free API.

### 9. Undo restores swallow failures

**What:** `UndoToast.onUndo` does `await restoreCleanupEntry(...).catch(() => {})` per entry, then closes the toast and refreshes — reporting implicit success even if every restore failed (recycle bin emptied, drive detached).

**Fix:** count failures and surface them:

```ts
const failures: string[] = [];
for (const id of undo.entryIds) {
  try { await restoreCleanupEntry(indexPath, id); }
  catch (e) { failures.push(String(e)); }
}
if (failures.length) setError(`${failures.length} of ${undo.entryIds.length} items could not be restored`);
else setUndo(null);
```

(Same pattern already exists in LibraryOverlay's per-row restore — reuse it.)

### 10. Retention constants drift

`workspace/.../ReviewModal.tsx` hardcodes `RETENTION_DAYS = 30`; `executor.rs` says `DEFAULT_RETENTION_DAYS = 90`. Works today because the frontend always passes 30 explicitly, but the backend default and every UI string will drift apart the first time someone adds a call site. **Fix:** make the backend default 30 to match the product copy, and have the frontend *omit* the parameter (single source of truth in Rust).

### 11. Lens payload is all 124k folders on every refresh

**What:** `treemap_lens_data` returns one row per folder in the index; the frontend builds a 124k-entry `Map` in React state and re-fetches it after every scan/clean/enrichment refresh. That's ~15–20 MB of JSON over IPC per refresh, plus GC churn.

**Impact:** multi-second refreshes and memory pressure that grow linearly with index size; the treemap only ever displays ~top-4000 folders (`QUERY_LIMIT`).

**Fix:** filter server-side to match what the UI can consume — e.g. `WHERE fo.total_bytes >= ?` or top-N by bytes with N = the overview query limit — and add `folder_path` to a shared threshold so `lensByPath` stays consistent with the overview folders. One `HAVING` clause; the Rust rollup loop stays identical.

### 12. Heuristics populator quadratic scans

**What:** `heuristics.rs::load_backup_origin_candidates` runs `lower(f.name) LIKE ?` over the whole `files` table (734k rows) once per backup-named file. This is why even "cheap" enrichment exceeded 10 minutes on the real index. (Already flagged as a spawned follow-up task.)

**Fix direction:** build one in-memory `HashMap<normalized_stem, Vec<FileLite>>` in a single pass, then match backups against it; or add a `lower(name)` expression index. Target: cheap enrichment ≤ 60 s on the 734k-file index.

### 13. Job-event double-apply

**What:** `useScanJob` attaches a live listener *and* backfills `nativeJobEvents(jobId, 0)`. Events that arrive on both paths append duplicate log lines (`apply` is not idempotent for `log_line` events).

**Fix:** number events server-side (index in the buffer) and track `lastApplied` client-side, skipping `n <= lastApplied`. The backfill already has offsets; the live event just needs to carry its index.

### 14. `api.rs` god module (SRP)

56 KB / ~1,700 lines mixing DTO definitions, overview queries, search, duplicates, cleanup, ontology surface, saved views, and OS shell integration. Every feature touches this file; merge conflicts and accidental coupling follow. **Fix:** mechanical split into `native/{dto,query,cleanup,ontology,shell}.rs` — no behavior change, do it before the next feature lands in it (M10 touches duplicates, which lives there).

## P3 (tracked debt)

15. **`IndexWriter` is also the reader** — `largest_folders`, `duplicate_groups` etc. live on a struct named and documented as a writer, and opening one runs migrations even for pure reads. Extract an `IndexStore` read facade over the same connection helper (#5) when convenient.
16. **Single `WorkspaceContext`** — every state change re-renders all consumers. Fine at current component count; if profiling shows selection-change jank on big treemaps, split selection/tray/nav contexts rather than adding memo everywhere.
17. **Dead commands registered** — `scan_to_index`, `trash_files` (see #3), and `duplicate_group_files`/`search_files` wrappers unused by the new UI (duplicates return in M10 — keep those two).
18. **Poisoned-lock handling** — `push_event` does `if let Ok(mut jobs) = jobs.lock()`, silently dropping events after a worker panic; job then looks "running" forever. Prefer `lock().unwrap_or_else(|p| p.into_inner())` (the data is a plain event buffer; poisoning adds nothing here).
19. **No `React.StrictMode`**, and the Inspector/UndoToast key listeners re-subscribe every render (no deps array). Both harmless today; StrictMode would have caught the double-apply in #13 earlier.
20. **`FINDING_KINDS` hardcodes two discovery kinds** — adding a discovery type means editing the Board. Acceptable now; when kinds grow, list them from a backend `discovery_kinds()` call instead.

Also verified and fine: SQL injection (all queries parameterized), path traversal in `delete_index` (canonicalize + prefix check), enrichment resumability after `kill` (populator state committed incrementally), scanner symlink handling (per README; not re-audited in depth this pass).

## Feature-gap root causes (from the 2026-07-03 retest)

**Board is empty** — two causes. (a) The Board surfaces only two discovery kinds (`derivedFrom-pattern`, `backupOf-pair`), which are *file-stem* heuristics (`report.bak` ↔ `report`), not folder-level ("Old HDD - Backup") relationships. (b) The redundancy that actually dominates this drive — **duplicates** — is computed by the scan (`duplicate_groups`, 0.80+ confidence, already in the overview payload) but the new UI never displays it anywhere. The old frontend's dedup viewer was its main consumer.

**Duplicates / file-level depth / preview / reveal / move suggestions** — the backend for duplicates (`duplicate_group_files`), reveal (`reveal_in_explorer`), and per-file provenance already exists and is exported in the bridge; only UI was lost with the old frontend. Reveal is restored as of this audit (Inspector → "⌖ Reveal"). Preview requires re-enabling the asset protocol *scoped* (see finding #2) — do not restore it with `**`. "Move/consolidate" suggestions were never in the Wave 1 cleanup vocabulary (it trashes, it doesn't relocate); that's the Wave 2 "Consolidatable Manifestations" concept — roadmap, not regression.

**Enrichment popup hides when a scan loads** — by design, but underexplained: the prompt is a *per-index opt-in* and unmounts once intelligence is enabled for the loaded index (your screenshots show verdicts, i.e. it's enabled) or after "Not now". The real gap is that there's no explicit "re-run enrichment" control afterwards — today the answer is "run a scan" (phase 2 re-enriches). M10 adds a Settings row for it.

### Proposed M10 (restore the lost surfaces)

1. **Duplicates surface** — a "Duplicates" section on the Board (or a Results saved-view chip): list `duplicate_groups` from the overview, drill into `duplicate_group_files`, stage members directly to the Cleanup Tray, per-file Reveal. ~1 day.
2. **File preview pane** — Inspector preview for images/video using the asset protocol scoped to scanned roots only. ~½ day after #2's scoping work.
3. **Settings: "Re-run enrichment"** — one row that enqueues the incremental-rescan job. ~1 hour.
4. **Audit fixes P0/P1** — CSP, asset scope, command removal, connection helper, executor ordering, event-buffer cap. ~1 day.

Recommended order: 4 → 1 → 3 → 2.

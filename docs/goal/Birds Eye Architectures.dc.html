What each contributes
AShell — the permanent frame: rail, tree, inspector, tray. Nothing reloads.
BBoard — the stage's most powerful lens; every answer can persist as a linked card.
CCommand — the spine that issues verbs & questions to whatever lens is open.
The glue — why no context-switching
Two things are global and shared by all three lenses: selection (drives the one Inspector) and the cleanup tray (collects staged items from anywhere). So flipping Treemap → Board → Results never loses your place — the right rail and bottom tray stay put while only the stage changes how it draws.
Workflow — one continuous loop
Scan — pick scope in tree, or type a command.
Explore — Treemap lens for size-first browsing.
Discover — command "relate" → result pins to Board.
Evaluate — shared Inspector judges safety.
Clean — shared tray, from any lens.
Learn — Board + history keep the reasoning.
Navigation model — lenses, not pages
The stage has exactly one switcher: Treemap · Board · Results. It changes the representation of the current scope — not the location. Cross-lens verbs stitch them together: send to Board, open as Treemap, pin result, focus inspector. The command bar (⌥⌘K) and history stack are the universal "address bar" + back button; the minimap handles long-range board moves. You never load a page — you change how you're looking or what you're asking.
Why this beats picking one
＋ Browsers get the Treemap, investigators get the Board, experts get the Command bar — same product, no compromise. Findings persist; cleanup is one shared tray.
－ More to build & more to teach — needs a clear default lens per scope and restrained cross-lens verbs so it doesn't feel like three apps bolted together.

Q · the command bar
Does a command change the board, or just link to it?
Visually linked — never a mutation. Running a command renders its answer as a result card pinned to the Board and draws edges to the cards it relates to. It's a live, re-runnable query view, not an edit to your files. The only thing that ever touches disk is staging an item into the tray and clearing the Review gate above. Commands explore; the gate acts.
Q · the lens switcher
So the treemap and board are switchable?
Yes — they're two lenses on the same scope. The switcher (Treemap · Board · Results) only changes how the stage draws the current selection; it doesn't navigate anywhere. Your selection, the Inspector, and the Cleanup Tray all persist across the flip — so you can browse sizes in Treemap, hit one key, and the same data becomes a board you can annotate and link.

Information architecture
Workspace (single) ├ Activity rail (modes) │ ├ Treemap ├ Scan │ ├ Findings ├ Saved views │ └ Timeline ├ Scope tree (left) ├ Visualization (center) ├ Inspector (right, dock) └ Cleanup tray (bottom)
Navigation model
No page loads. The shell is permanent; the rail swaps the center mode while tree + inspector persist. Drill-down zooms the treemap in place; breadcrumb tracks depth. Panels are dockable, collapsible, and resizable — like an IDE. Everything is one keystroke or one click away from everything else.
Workflow walkthrough
① Rescan from toolbar — tree + map repaint.
② A big rectangle catches the eye → click to drill.
③ Inspector explains why it exists + relations.
④ Stage reclaimable items into the tray.
⑤ Review & clean → loop repeats, map shrinks.
Component hierarchy
<Workspace> → <ActivityRail> · <ScopeTree> · <CenterStage mode="treemap|scan|findings|timeline"> → <TreemapNode>* · <Inspector> → {<WhatPanel>, <WhyPanel>, <RelationsList>, <SafetyVerdict>, <ActionBar>} · <CleanupTray> → <StagedItem>*. The Inspector is a single component bound to "current selection" — selection is global workspace state, so any surface can drive it.
Trade-offs
＋ Zero context-switching; familiar to devs; dense.
－ Panel-heavy can feel busy; weaker for non-linear "how does this relate" exploration; treemap-first biases toward size over meaning.

Information architecture
Boards (many) └ Board = investigation ├ Cards (free-placed) │ ├ Treemap │ ├ Relation graph │ ├ Finding / note │ └ File list ├ Links between cards └ Staged items Palette · Search · Minimap
Navigation model
Pan & zoom replace pages — spatial memory is the navigation. Drag from the palette to spawn cards; drag between cards to link evidence. Search drops a result card onto the board. The minimap and board switcher handle long-range jumps. Nothing is ever "closed" — you zoom away and it persists.
Workflow walkthrough
① Drop a question card to frame the scan.
② Pull in a treemap of the suspect scope.
③ Expand a relation graph — links reveal shared data.
④ Capture the conclusion as a finding card.
⑤ Stage & clean straight from the finding.
Component hierarchy
<BoardCanvas pan zoom> → <Card kind="treemap|graph|finding|note|list">* · <Link from to>* · <Palette> · <Search> · <Minimap> · <BoardRail>. Every card exposes the same <ActionBar> (stage / dismiss / link), so cleanup is available wherever attention already is. Cards are self-contained queries — re-running a scan repaints them in place.
Trade-offs
＋ Best-in-class for "how things relate"; reasoning persists; delightful.
－ Freeform = a blank-canvas cold start; spatial sprawl needs discipline; less efficient for a quick "just delete the obvious".

keyboard-first · a query bar drives everything

Information architecture
Command bar (always primary) ├ Verbs │ scan · find · explain │ relate · compare │ stage · clean ├ Result surfaces │ ├ Icicle chart │ └ Ranked list (actionable) ├ History stack (notebook) └ Pinned commands
Navigation model
There are no pages to navigate — you compose queries. ⌥⌘K focuses the bar from anywhere; ↵ runs, ⇧↵ stages, ⌘↵ cleans. The history stack is the back button: every command is a permalink you can rerun or pin. Results are ephemeral views generated on demand, never destinations you "go to".
Workflow walkthrough
① Focus the bar, type a plain-language ask.
② Autocomplete resolves it to a verb + scope.
③ Icicle + ranked list render the answer.
④ Stage safe rows without leaving results.
⑤ Clean; the command joins history to rerun.
Component hierarchy
<CommandBar> → <VerbResolver> · <SuggestionList> · <ResultView> → {<IcicleChart>, <RankedList> → <ResultRow safety stageable>*} · <HistoryStack> → <CommandChip rerun pin>* · <StatusBar stage clean>. Every query is a serializable object {verb, scope, predicate, sort} — which makes commands pinnable, rerunnable, and scriptable.
Trade-offs
＋ Fastest for experts; near-zero chrome; fully keyboard; scriptable.
－ Steep cold start; weak for serendipitous browse & spatial reasoning; relies on a strong query language to feel magical.
